# Lab1：Ping-Pong 进程间通信实验
## 实验目标
实现一个 `ping-pong` 程序，在父子两个进程之间循环发送一个字节，测试一秒内两个进程直接循环发送的总字节数。
允许使用的系统调用：`pipe`、`fork`、`read`、`write`

---

## 核心设计思路
1.  **进程创建**：通过 `fork()` 创建子进程，父进程与子进程通过管道通信。
2.  **双管道设计**：
    - 管道是半双工的，一个管道只能固定一端读、一端写，因此需要**两个管道**实现双向通信：
      - `p1`：父进程写 → 子进程读
      - `p2`：子进程写 → 父进程读
    - 避免单管道同时读写带来的逻辑混乱和性能损耗。
3.  **循环逻辑**：父子进程形成「父写→子读→子写→父读」的循环，统计1秒内传输的总字节数。

---

## 完整实现代码
```C
#include "kernel/types.h"
#include "user/user.h"

int main(int argc, char* argv[]) {
    // 两个管道：p1[0]读/[1]写（父写子读），p2[0]读/[1]写（子写父读）
    int p1[2], p2[2];
    pipe(p1);
    pipe(p2);

    int pid = fork();
    if (pid < 0) {
        fprintf(2, "fork error\n");
        exit(1);
    }

    // -------------------------- 子进程 --------------------------
    if (pid == 0) {
        // 关闭子进程不需要的文件描述符：p1的写端、p2的读端
        close(p1[1]);
        close(p2[0]);

        char buf;
        uint64 start_time = uptime(); // xv6系统时间，单位为ticks
        uint64 count = 0;

        // 循环读写：子进程先读（从p1）再写（到p2）
        while(1) {
            // 从父进程读取1字节
            read(p1[0], &buf, 1);
            count++;

            // 写回给父进程
            write(p2[1], &buf, 1);
            count++;

            // 运行1秒后退出循环（xv6的ticks约为100/秒）
            if (uptime() - start_time >= 100) {
                break;
            }
        }

        // 关闭剩余文件描述符
        close(p1[0]);
        close(p2[1]);
        exit(0);
    }

    // -------------------------- 父进程 --------------------------
    else {
        // 关闭父进程不需要的文件描述符：p1的读端、p2的写端
        close(p1[0]);
        close(p2[1]);

        char buf = 'A'; // 初始传输的字节
        uint64 start_time = uptime();
        uint64 count = 0;

        // 循环读写：父进程先写（到p1）再读（从p2）
        while(1) {
            // 发送1字节给子进程
            write(p1[1], &buf, 1);
            count++;

            // 接收子进程写回的字节
            read(p2[0], &buf, 1);
            count++;

            // 运行1秒后退出循环
            if (uptime() - start_time >= 100) {
                break;
            }
        }

        // 关闭文件描述符
        close(p1[1]);
        close(p2[0]);

        // 回收子进程资源
        wait(NULL);

        // 打印1秒内传输的总字节数
        printf("Total bytes transferred in 1 second: %lu\n", count);
        exit(0);
    }
}
```
## 遇到的问题以及解决方案
### 1. 为什么放弃「单管道双向读写」方案？
我最初考虑过只用一个管道实现双向通信，但分析后放弃了这个方案，原因如下：
竞态问题：父子进程同时对一个管道进行读写，会出现「自己读到自己写的数据」的情况，需要额外的标志位来区分消息来源，实现复杂。
性能损耗：标志位校验和锁机制会增加额外开销，远不如双管道的直接读写高效。
代码可读性差：单管道的读写逻辑耦合度高，后期维护和调试难度大。
因此，最终选择了双管道方案，虽然多了一个管道，但逻辑清晰、无竞态、性能最优。
### 2. 代码中的细节问题修正
初始代码中childbuf[1024] = "";的初始化方式不规范，改为单字节char buf，避免多余的 IO 开销。
补充了fork()错误处理和文件描述符关闭的顺序，避免管道阻塞。
增加了wait(NULL)回收子进程资源，避免僵尸进程。
